2. Аналитик хочет иметь систему со следующими возможностями:

2.1. Система должна хранить события, которые в последующем будут отправляться сервису событий
2.2. События характеризуются важностью (аналитик готов выставлять важность в целых числах)
2.3. События характеризуются критериями возникновения. Событие возникает только если выполнены все критерии его возникновения. Для простоты все критерии заданы так: <критерий>=<значение>

Написать систему, которая будет уметь:
    - добавлять новое событие в систему хранения событий
    - очищать все доступные события
    - отвечать на запрос пользователя наиболее подходящим событием
    - использовать для хранения событий redis

--------------------

События храним в хеш таблице

events
    event_name {event:object}

--------------------
API

Получить все события
POST /api/events

Добавление события
POST /api/event/add
{
    priority: 1000,
    conditions: {
        param1: 1
    },
    event: {
        name: event_1
    },
}

Удалить событие
POST /api/event/remove
{
    "key": "xxx"
}

Удалить все события
POST /api/event/clear

Запрос от пользователя
POST /api/request
{
    params: {
        param1: 1,
        param2: 2
    }
}

#######################

Таким образом предположим, что аналитик заносит в систему следующие события:
{
    priority: 1000,
    conditions: {
        param1: 1
    },
    event: {
        name: event_1
    },
},
{
    priority: 2000,
    conditions: {
        param1: 2,
        param2: 2
    },
    event: {
        name: event_2
    },
},
{
    priority: 3000,
    conditions: {
        param1: 1,
        param2: 2
    },
    event: {
        name: event_3
    },
},

От пользователя приходит запрос:
{
    params: {
        param1: 1,
        param2: 2
    }
}
